Auth And Session Architecture
- Session model
  - One secure httpOnly session cookie: `zc_session`
  - Preference cookies: `zc_theme`, `zc_locale`
  - Cookie keys are defined in `packages/shared-config/src/env.ts`
- Active auth flow
  - Auth.js credentials providers (`user-credentials`, `admin-credentials`) are active in `apps/web/auth.ts`
  - Clients submit Supabase access tokens (`idToken`) to Auth.js handlers
  - Server verifies tokens through `getServerAuthAdmin().verifyIdToken(...)`
  - Session strategy is Auth.js JWT with server-side callback normalization
- Endpoint ownership
  - `GET|POST /api/auth/[...nextauth]` is the active auth handler surface
  - `/api/auth/bootstrap` and `/api/auth/admin/bootstrap` are deprecated endpoints that return 410 migration signals
  - `/api/auth/me` remains the app-specific authenticated session/runtime status endpoint
- Route protection
  - `apps/web/proxy.ts` remains optimistic redirect support only
  - authoritative guards remain server-side via session helpers (`requireAuthenticatedUser`, `requireCompletedUser`, `requireAdminUser`)

Admin Login Architecture
- Admin authority inputs
  - allowlisted admin emails from `ZOOTOPIA_ADMIN_EMAILS`
  - supported username aliases resolved by `POST /api/auth/admin/resolve-identifier`
- Admin session issuance requirements (active flow in `apps/web/auth.ts`)
  - valid/recent Supabase access token
  - email/password provider identity (`password` provider)
  - allowlisted email

2026-04-11 - Auth callback root-cause isolation, auth-critical Supabase-only persistence, and stage tracing

Root Cause (confirmed with runtime evidence)
- Exact failing statement in the pre-fix callback path was the users read inside `getUserByUid(...)`:
  - `await getZootopiaFirestore().collection("users").doc(uid).get();`
  - function: `getUserByUid(...)` in `apps/web/lib/server/repository.ts`
- Exact thrown error from direct DB probe and callback-linked path:
  - `42P01 relation "zc_entities" does not exist`
- Why it failed:
  - Auth callback `authorizeAdminCredentials(...)` and `authorizeUserCredentials(...)` both call `upsertUserFromAuth(...)`.
  - `upsertUserFromAuth(...)` called `getUserByUid(...)`, which depended on the `zc_entities` table through the Firestore-shaped Postgres adapter.
  - Missing table caused callback-runtime exception, which was normalized to `ADMIN_BOOTSTRAP_FAILED` / `BOOTSTRAP_FAILED`, then session bootstrap ended with `SESSION_NOT_ESTABLISHED`.

- `apps/web/lib/server/auth-tracing.ts` (new)

- `apps/web/lib/server/repository.ts`

  - `createAuthTraceContext(...)`

  - `logAuthStageFailure(...)`
  - `classifyAuthError(...)`
- Instrumented callback/token/session paths in `auth.ts`:
  - runtime, token-required, token-verify, recent-signin, provider-check, admin-allowlist, admin-claim, user-upsert, status-check, jwt-callback, session-callback, signIn-event.
- Instrumented `server-auth.ts` token verification and user lookup.
- Instrumented `admin-auth.ts` identifier allowlist resolution and claim activation checks.
- Removed auth-critical dependency on Firestore-shaped `zc_entities` path in `repository.ts`:
  - `getUserByUid(...)` now resolves from Supabase Auth via `getServerAuthAdmin().getUser(...)`.
  - `upsertUserFromAuth(...)` now persists auth-critical user metadata as Supabase app_metadata claims (`setCustomUserClaims`) through `persistAuthCriticalUserClaims(...)`.

2026-04-11 | Migration-First Database System Implementation

Root Cause
- Error: `relation "zc_entities" does not exist` at `PgQuery._getWithSql`
- Cause: Migration files existed in `supabase/migrations/` but had NEVER been applied
  to the actual Supabase Postgres database. The code assumed the table existed at
  runtime, but no migration runner had been executed.
- `zc_entities` IS the intended main table — it is a generic JSON document store
  (EAV pattern) used by the Postgres adapter for all logical collections
  (documents, assessments, infographics, credits, idempotency, admin logs, users).

Whether zc_entities Was Truly Intended
- YES. It is intentionally designed as a generic JSONB document store that mirrors
  former Firestore collections. The `collection` column acts as a namespace, `id` as
  the document key, `owner_uid` as the indexed owner, and `body` as the full JSONB
  payload. This is the active persistence model for the current codebase.

How to Bootstrap a Fresh Database from Zero
1. Ensure `DATABASE_URL` or `SUPABASE_DATABASE_URL` is set in `.env.local`
2. Run: `npm run db:apply`
3. All pending migrations are applied in order inside transactions
4. Verify: `npm run db:apply:dry` should report "All migrations are already applied"

Local Workflow Commands
- `npm run db:apply` — Apply all pending migrations
- `npm run db:apply:dry` — Preview what would be applied (no changes)
- `npm run supabase:migrations:list` — List migration files
- `npm run supabase:migrations:validate` — Validate migration file ordering
- `npm run supabase:check` — Validate + list

Production Workflow Commands (Vercel + Supabase)
- Migrations are NOT auto-applied on deploy. They must be run manually or via CI/CD.
- Option A (manual): Run `node scripts/supabase/apply-migrations.mjs` with production
  DATABASE_URL set (use Vercel CLI env pull or run locally with prod connection string)
- Option B (CI/CD): Add a pre-deploy step in your CI pipeline that runs the migration
  script against the production database before deploying the app
- Option C (Supabase Dashboard): Paste migration SQL into Supabase SQL editor manually
- IMPORTANT: Always run migrations BEFORE deploying code that depends on new tables

Migration System Design
- Script: `scripts/supabase/apply-migrations.mjs`
- Uses `postgres` npm package (already installed)
- Loads `.env.local` via dotenv for standalone operation
- Creates `_migration_history` table to track applied migrations
- Each migration runs inside a transaction (atomic)
- Failed migrations stop execution and report the error
- Idempotent: re-running skips already-applied migrations
