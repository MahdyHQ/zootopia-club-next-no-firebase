4. ARCHITECTURE STYLE
================================================================================

Pattern:        Next.js App Router with layered server architecture
Style:          Monorepo with workspace isolation
Session:        Stateless JWT (Auth.js / NextAuth v5)
Database:       Supabase Postgres (provider-managed)
Storage:        Supabase Storage buckets (provider-managed)
AI:             Multi-provider orchestration (Google AI + DashScope)

Layered Architecture:
```
Route Handlers (apps/web/app/api/)
  → Session/Auth Layer (lib/server/session.ts)
  → Repository Layer (lib/server/repository.ts)
  → AI Execution Layer (lib/server/ai/execution.ts)
  → Storage Layer (lib/server/supabase-blob-storage.ts)
  → Database Layer (lib/server/zootopia-postgres-adapter.ts)
```

Boundaries:
- Frontend: Client components in (site), (public), (protected)/(completed)
- Backend: Route handlers, lib/server/*, auth.ts, proxy.ts
- Shared: packages/shared-* (types, config, utils)
- Infrastructure: supabase/migrations/, scripts/, vercel.json
- Provider-Managed: Supabase (Auth, Postgres, Storage), Vercel (Hosting), Resend (Email)

Same-Origin vs Provider-Managed:
- Next.js app runs on Vercel (same-origin for all /api/* routes)
- Supabase Auth: cross-origin provider (browser → Supabase CDN)
- Supabase Postgres: server-side only (connection string, not browser-accessible)
- Supabase Storage: cross-origin provider (browser + server)
- Google AI API: cross-origin provider (server-side calls)
- DashScope API: cross-origin provider (server-side calls)
- Resend/SMTP: server-side only

12. AUTH & SESSION ARCHITECTURE
================================================================================

Framework: Auth.js (NextAuth v5)
Strategy: JWT (stateless sessions)
Identity Provider: Supabase Auth (email/password only)

Credential Providers:
1. user-credentials — Regular user login
2. admin-credentials — Admin login (separate provider)

User Login Flow:
1. Client signs in via Supabase browser client → receives idToken
2. Client calls signIn("user-credentials", { idToken, ...deviceMetadata })
3. Server (auth.ts → authorizeUserCredentials):
   a. Runtime check: Supabase admin runtime + allowlist must be configured
   b. Rate limit: 20 requests/minute per IP
   c. Token verification: verifySupabaseAccessToken() via Supabase getUser()
   d. Recency check: auth_time must be within 5 minutes
   e. Admin block: If email in ZOOTOPIA_ADMIN_EMAILS, reject with ADMIN_LOGIN_REQUIRED
   f. Provider check: Must be password provider
   g. User upsert: upsertUserFromAuth() creates/updates profile in Postgres
   h. Status check: User must be active (not suspended)
4. JWT callback populates token with uid, role, status, profile completion
5. Session callback syncs token to session object
6. Client receives session cookie

Admin Login Flow:
1. Admin signs in via Supabase browser client
2. Client calls signIn("admin-credentials", { idToken })
3. Server checks email against ZOOTOPIA_ADMIN_EMAILS allowlist
4. If not in allowlist, reject
5. If in allowlist, proceed with same verification as user flow
6. JWT populated with role: "admin"

Session Configuration:
- Strategy: JWT (no database sessions)
- Max age: ZOOTOPIA_SESSION_TTL_SECONDS (default 3600 = 1 hour)
- Cookie: httpOnly, secure, sameSite: lax

Key Files:
- apps/web/auth.ts — Auth.js configuration
- apps/web/proxy.ts — Auth.js proxy helpers
- apps/web/lib/auth-*.ts — Auth utilities
- apps/web/lib/server/session.ts — Server session utilities

14. PROFILE COMPLETION & USER GATING MODEL
================================================================================

Profile Completion Gating:
- Users must complete their profile before accessing protected features
- Incomplete profiles are redirected to /settings to finish setup
- Profile completion status stored in JWT session token
- Gating applied at:
  - Route level (middleware or layout guards)
  - API level (server-side checks in route handlers)

Gating Rules:
1. User logs in → session created
2. Session checked for profile completion
3. If incomplete → redirect to /settings
4. If complete → access to protected routes
5. Admin users bypass profile completion gating

Profile Fields (typical):
- Display name
- Phone number (validated with libphonenumber-js)
- Preferences (theme, locale)
- Any required onboarding fields

Implementation:
- Profile completion check in session token
- Server-side guards in API routes
- Client-side guards in protected layouts
- Redirect logic in auth flow

16. ASSESSMENT RUNTIME / RESULT / EXPORT FLOW
================================================================================

Assessment Generation Flow:
1. User navigates to /assessment
2. User selects document and configures assessment options
3. Client sends POST /api/assessment
4. Server performs checks:
   a. Authentication check
   b. Profile completion check
   c. Admin bypass (if admin)
   d. Idempotency check
   e. Credit reservation check (daily limit)
5. AI generation via ai/execution.ts
6. Assessment result saved to Postgres
7. Admin log entry created
8. Response includes assessment data

Credit System:
- Daily credit limit: ZOOTOPIA_DEFAULT_DAILY_ASSESSMENT_CREDITS (default 3)
- Credits reset daily
- GET /api/assessment/credits returns current status
- Credit reservation before AI generation
- Credit consumed on success, released on failure

Assessment Retrieval:
- GET /api/assessment/[id] — Single assessment
- GET /api/assessments — List user assessments
- GET /api/assessment/linked-document — Resolve linked document context

Export Flow:
- GET /api/assessment/export/* — Export assessment
- Formats: PDF, JSON
- Locale and theme variants supported
- PDF generation uses Puppeteer (puppeteer-core + @sparticuz/chromium)
- Export retention: 15 minutes (configurable)

Assessment Retention:
